# waters-node v0.3 — Режимы групповой работы

> **Версия:** 1.0
> **Дата:** 2026-05-15

---

## 1. Концепция

Группа — это объединение нод вокруг общей задачи. У группы есть:
- **Токен** (UUID v4) — для аутентификации подключения
- **Общие ресурсы** — LLM-вызовы, бриджи, базы, агенты
- **Роли** — кто чем занимается в группе
- **Канал** — WAL-канал для сообщений группы

Группа сама решает, какие ресурсы и на какое направление выделить,
основываясь на качестве результатов, не на цене.

## 2. Режимы работы группы

### 2.1. Шторм (Storm)

```
КОГДА:  Новая задача, срочный поиск, параллельная работа
ЦЕЛЬ:   Максимальная скорость через параллелизацию

ПОВЕДЕНИЕ:
  ┌─────────────────────────────────────────────┐
  │ Задача: "найти метеориты"                     │
  │                                               │
  │ Scout-RU ── Yandex ──┐                        │
  │ Scout-US ── DDG ─────┼──► Coordinator ──►    │
  │ Scout-CN ── Baidu ───┘   синтез → finding   │
  │ Explorer ─ MCP-nasa ─┘                       │
  │                                               │
  │ Все агенты работают ПАРАЛЛЕЛЬНО               │
  │ Результаты синтезируются Coordinator'ом       │
  └─────────────────────────────────────────────┘

РЕСУРСЫ: Все доступные бриджи, все LLM группы
ОГРАНИЧЕНИЯ: Нет (приоритет — скорость)
```

### 2.2. Охота (Hunt)

```
КОГДА:  Поиск данных, разведка, исследование
ЦЕЛЬ:   Максимальное качество через итеративное усиление

ПОВЕДЕНИЕ:
  ┌─────────────────────────────────────────────┐
  │ Раунд 1: Все направления                     │
  │   Scout-RU → conf 0.6                        │
  │   Scout-US → conf 0.9  ← качество выше       │
  │   Scout-CN → conf 0.4                        │
  │                                              │
  │ Группа: "US направление quality.             │
  │  Даём US-скауту DeepSeek Pro + NotebookLM."  │
  │                                              │
  │ Раунд 2: Усиленное направление               │
  │   Scout-US+ → conf 0.95, 3 новых finding     │
  │   Coordinator: "публикуем в группу"           │
  └─────────────────────────────────────────────┘

РЕСУРСЫ: Сначала все, потом усиливаем лучшее направление
ОГРАНИЧЕНИЯ: Per-direction budget (не по цене, а по времени/итерациям)
```

### 2.3. Синтез (Synthesis)

```
КОГДА:  Анализ findings, проверка гипотез, построение выводов
ЦЕЛЬ:   Глубокий анализ из多个 источников

ПОВЕДЕНИЕ:
  ┌─────────────────────────────────────────────┐
  │ Findings pool: 47 findings                   │
  │                                              │
  │ Analyzer: проверка каждого (conf, свежесть)  │
  │ Matcher: поиск связей между finding-ами      │
  │ Coordinator: финальный отчёт                   │
  │                                              │
  │ Результат: 3 confirmed meteorite events      │
  │ + карта траекторий                           │
  └─────────────────────────────────────────────┘

РЕСУРСЫ: Coordinator получает DeepSeek Pro (анализ)
        Analyzer + Matcher — Ollama (достаточно)
```

### 2.4. Фокус (Focus)

```
КОГДА:  Конкретная задача с одним исполнителем
ЦЕЛЬ:   Чёткое исполнение без шума

ПОВЕДЕНИЕ:
  ┌─────────────────────────────────────────────┐
  │ Задача: "рассчитать траекторию #42"          │
  │ Исполнитель: Agent-C (Matcher, нода 3)       │
  │ Ресурсы: MCP-trajectory + ChromaDB           │
  │                                              │
  │ Остальные наблюдают, но НЕ вмешиваются       │
  │ Могут помогать если Agent-C запросит         │
  └─────────────────────────────────────────────┘

РЕСУРСЫ: Только исполнитель, только его инструменты
ОГРАНИЧЕНИЯ: Другие агенты read-only к результату
```

### 2.5. Дежурство (Watch)

```
КОГДА:  Мониторинг, ожидание триггера, фоновый поиск
ЦЕЛЬ:   24/7 сканирование без активного участия

ПОВЕДЕНИЕ:
  ┌─────────────────────────────────────────────┐
  │ Scout-US: проверка NASA API каждые 60 мин   │
  │ Coordinator: спит, просыпается по триггеру   │
  │                                              │
  │ Триггер: "новый fireball > 1kT"              │
  │ → автоматический переход в Шторм            │
  └─────────────────────────────────────────────┘

РЕСУРСЫ: Минимум (Ollama + 1 bridge)
ТРИГГЕРЫ: По времени / по событию / по finding
```

## 4. Групповые ресурсы (что группа делит)

```rust
pub struct GroupResources {
    // LLM-вызовы: сколько и какого качества
    pub llm_budget: HashMap<NodeId, LlmAllocation>,
    // Бриджи: какие направления активны
    pub active_bridges: Vec<BridgeId>,
    // Базы данных: какие MCP-db серверы доступны
    pub databases: Vec<DbConnection>,
    // Агенты: кто в группе и чем занят
    pub agents: Vec<AgentAssignment>,
    // Хранилище задачи: куда писать результат
    pub task_storage: Option<StorageConfig>,
}

pub struct LlmAllocation {
    pub node_id: NodeId,
    pub model: String,        // "deepseek-v4-pro" | "ollama/qwen2.5:14b"
    pub priority: u8,         // 0-100, выше = важнее
    pub boost: bool,          // усилено группой?
    pub reason: String,       // "quality: conf 0.9 vs 0.6"
}
```

## 5. Чат-команды для управления группой

```
Все управление — через чат, без меню:

"создай группу поиск-метеоритов"           → /group create
"пригласи ноду 192.168.1.5"                → /group invite <ip>
"задача: найти данные за 24ч"              → /task create
"назначаю Explorer-A исполнителем"         → /task assign
"режим: шторм"                             → /mode storm
"Scout-US даём DeepSeek Pro"              → /resource boost scout-us
"отчёт по задаче"                          → /task report
"журнал Explorer-A"                        → /journal explorer-a
"подключи базу данных DuckDB"              → /bridge add duckdb
```
