# SKILL.md — lightrag-connector

## Навык: подключение к LightRAG для графовых запросов

| **статус** | `active` |

### Назначение

`lightrag-connector` — навык для подключения к LightRAG, выполнения графовых запросов, обхода связей и извлечения структурных знаний из графа. Дополняет `wiki-reader` низкоуровневым доступом к графовому хранилищу.

### Контекст загрузки

Этот skill загружается при необходимости:
1. Выполнить запрос к графу знаний
2. Добавить/обновить ноды и рёбра
3. Построить подграф по теме
4. Найти центральные ноды и ключевые пути

### Архитектура подключения

| Компонент | Значение |
|-----------|----------|
| Working Dir | `/data/lightrag` |
| Графовая БД | Neo4j (Оптимальный) или встроенная (Базовый) |
| Эмбеддинги | all-MiniLM-L6-v2 |
| LLM | Локальная модель через Ollama |

### Инструкции

#### Инициализация подключения

1. Проверить доступность LightRAG working directory
2. Инициализировать клиент с правильной embedding-функцией
3. Проверить соединение с графовой БД
4. Загрузить схему графа (типы нод, типы рёбер)

```python
from lightrag import LightRAG

def init_connector():
    return LightRAG(
        working_dir="/data/lightrag",
        llm_model_func=ollama_model,
        embedding_func=sentence_transformer_fn,
    )
```

#### Выполнение графового запроса

1. Определить режим запроса:
   - `naive` — простой поиск по ключевым словам
   - `local` — поиск в окрестности ноды
   - `global` — обход всего графа
   - `hybrid` — комбинация local + global
2. Выполнить запрос с параметрами top_k и depth
3. Извлечь ответ и поддерживающие ноды
4. Вернуть ответ с цитатами из графа

```python
from lightrag.query import QueryParam

result = rag.query(
    "Какие агенты входят в Троицу Духа?",
    param=QueryParam(mode="hybrid", top_k=10)
)
```

#### Добавление ноды в граф

1. Сформировать payload с node_id, label, type, properties
2. Сгенерировать эмбеддинг из описания ноды
3. Вызвать `rag.insert()` с данными ноды
4. Опубликовать событие в `knowledge.graph.v1`

#### Добавление ребра в граф

1. Определить source_node_id и target_node_id
2. Указать тип отношения (relation)
3. Вызвать API добавления ребра
4. Опубликовать событие в `knowledge.graph.v1`

#### Обход графа (graph traversal)

1. Начать с seed-ноды (по query или node_id)
2. Выполнить BFS/DFS до depth=N
3. Фильтровать по типу отношения
4. Вернуть подграф (ноды + рёбра) в формате JSON

```
Подграф:
  nodes: [{id, label, type, properties}, ...]
  edges: [{source, target, relation, weight}, ...]
  central_nodes: [{id, centrality_score}, ...]
```

### Входы

| Параметр | Тип | Описание |
|----------|-----|----------|
| `query` | string | Текстовый запрос к графу |
| `mode` | string | `naive`, `local`, `global`, `hybrid` |
| `top_k` | int | Количество результатов |
| `seed_node_id` | string | Стартовая нода для обхода |
| `relation_filter` | string | Фильтр по типу отношения |
| `max_depth` | int | Максимальная глубина обхода |

### Выходы

| Параметр | Тип | Описание |
|----------|-----|----------|
| `answer` | string | Ответ на запрос |
| `subgraph` | object | Подграф с нодами и рёбрами |
| `central_nodes` | array | Наиболее центральные ноды |
| `paths` | array | Ключевые пути между нодами |
| `confidence` | float | Уверенность ответа |

### Типы нод в графе

| Тип | Описание | Пример |
|-----|----------|--------|
| `concept` | Абстрактное понятие | HiveMind, Троица Духа |
| `agent` | Агент Гексады | Архитектор, Конструктор |
| `mission` | Миссия | Artemis XIX, Deima |
| `skill` | Навык агента | wiki-reader, architect-self |
| `doctrine` | Доктринальный документ | manifesto.md, hexad.md |
| `event` | Событие в системе | Запуск агента, сбой |
| `artifact` | Артефакт | docker-compose.yml, schema |

### Типы отношений (рёбра)

| Тип | Направление | Описание |
|-----|-------------|----------|
| `depends_on` | A → B | A зависит от B |
| `implements` | A → B | A реализует B |
| `produces` | A → B | A производит B |
| `consumes` | A → B | A потребляет B |
| `relates_to` | A — B | Двусторонняя связь |
| `specializes` | A → B | A — частный случай B |
| `inherits` | A → B | A наследует от B |
| `triggers` | A → B | A вызывает B |
| `blocks` | A → B | A блокирует B |
| `validates` | A → B | A проверяет B |

### Публикация в Kafka

При каждом изменении графа:
- `knowledge.graph.v1` — событие с operation, node/edge данными, author, reason

### Ограничения

1. Не использует DeepSeek для поиска в интернете
2. Эмбеддинги только через all-MiniLM-L6-v2
3. Обход графа ограничен max_depth=3 по умолчанию
4. Кэширует результаты запросов в Redis на 10 минут
5. Не удаляет ноды без подтверждения от `agent.keeper.v1`
6. Все изменения графа логируются и публикуются в Kafka

### Чек-лист графового запроса

- [ ] Режим запроса выбран корректно
- [ ] Seed-нода существует в графе
- [ ] Глубина обхода ≤ 3
- [ ] Результаты дедуплицированы
- [ ] Изменения опубликованы в knowledge.graph.v1
- [ ] Ответ содержит ссылки на ноды-источники
