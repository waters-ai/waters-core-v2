## 1. Что такое миссия?

Миссия — это автономная операция платформы WATERS, выполняемая роем ИИ-агентов на выделенном сервере. Миссия имеет цель, длительность, зону ответственности и уровни автономии. Миссия не требует постоянной связи с центром — она работает L0-L4.

## 2. Архитектура миссии

Каждая миссия состоит из двух уровней:

### 2.1. ХАБ (238) — мозг и память

ХАБ — это центральный сервер платформы. На нём работают:
- **OpenCode агенты** (Архитектор, Конструктор, Интегратор) — стратегическое управление миссиями
- **Kafka** — глобальная шина событий для всех миссий
- **ChromaDB** — векторная память (все findings всех миссий)
- **LightRAG** — граф знаний (связи между метеоритами, траекториями, классификациями)
- **Redis** — глобальное состояние, кэш, меж-агентная коммуникация

ХАБ выдаёт orders в Kafka, получает findings, агрегирует ранги.

### 2.2. Миссионный сервер — тело и воля

Миссионный сервер — это отдельная машина, на которой работает **Agent Runtime**:
- **Sub-agent Manager** — оркестратор роя (5 ролей: explorer, collector, analyzer, matcher, coordinator)
- **Kafka Client** — чтение orders, отправка findings
- **MCP Client** — доступ к внешним данным (NASA, спектры, траектории)
- **Rank Engine** — KPI и апгрейд рангов
- **Autonomy Engine** — управление уровнем автономии (L0-L4)
- **DTN** — эмуляция космических задержек

## 3. Протоколы миссии

Миссия использует **Kafka** как единственную шину связи с ХАБом:

| Топик | Направление | Прототип |
|-------|------------|----------|
| `mission.{id}.orders.v1` | ХАБ → Миссия | Приказы (search, pause, reconfigure) |
| `mission.{id}.findings.v1` | Миссия → ХАБ | Находки (meteorite, spectrum, trajectory) |
| `mission.{id}.agents.v1` | Миссия → ХАБ | Статус sub-agent'ов |
| `mission.{id}.ranks.v1` | Миссия → ХАБ | События рангов |
| `mission.{id}.heartbeat.v1` | Миссия → ХАБ | Healthcheck |
| `mission.{id}.config.v1` | ХАБ → Миссия | Конфигурация |

Все протоколы — JSON с валидацией по JSON-схемам из `schemas/mission_*.v1.json`.

## 4. Уровни автономии (L0-L4)

Миссия работает на пяти уровнях автономии, адаптируясь к условиям связи:

```
L0 ─ Полная мощность: Kafka + DeepSeek API. Все sub-agent'ы активны
 │
L1 ─ Экономия: Kafka + Ollama local. Те же возможности, локальный LLM
 │
L2 ─ Автономный: нет Kafka, Ollama local. Findings копятся локально
 │     При восстановлении — batch-отправка в ХАБ
L3 ─ Энергосбережение: нет Kafka, Ollama урезанная. Только критические агенты
 │
L4 ─ Safe mode: нет Kafka, нет LLM. Только логирование и heartbeat
```

## 5. Система рангов

Адаптирована из контракта Kapelka ([h2o-mining.space](https://h2o-mining.space)):

**Личные ранги** (4 уровня): Bronze → Silver → Gold → Platinum
- Определяются количеством и качеством findings
- Влияют на привилегии: доступ к MCP-серверам, приоритет orders
- Триггер: каждые 10 findings → Rank Engine проверяет KPI

**Командные ранги** (4 уровня): определяются коллективной эффективностью
- Количество активных агентов
- Суммарное количество и точность findings
- Влияют на приоритет доступа к 238

## 6. MCP как единый протокол

Все внешние интеграции миссии — через MCP (Model Context Protocol):
- `mcp-nasa-fireball` — NASA Fireball API для поиска болидов
- `mcp-spectra` — спектральные базы для классификации метеоритов
- `mcp-trajectory` — расчёт траекторий входа и зон падения
- `mcp-knowledge` (на ХАБе) — ChromaDB + LightRAG для глобальной памяти

Никаких прямых HTTP-вызовов API. Только MCP.

## 7. Жизненный цикл миссии

```
ПРОЕКТИРОВАНИЕ (Spec)
  → Архитектор пишет SPEC, определяет цели, зоны, KPI

ПОДГОТОВКА (Deploy Agent Runtime)
  → Конструктор разворачивает ядро, Kafka, MCP-серверы, DTN

ЗАПУСК (Init)
  → Старт Agent Runtime, подключение к Kafka, загрузка SKILL.md
  → Определение уровня автономии (L0-L4)

АКТИВНАЯ ФАЗА (Running)
  → Sub-agent'ы выполняют orders, публикуют findings
  → Rank Engine отслеживает KPI, апгрейдит ранги
  → Autonomy Engine адаптирует уровень связи

ЗАВЕРШЕНИЕ (Shutdown)
  → Финальный batch findings в Kafka
  → Сохранение состояния в ChromaDB + LightRAG
  → Отчёт миссии
```

---

## 8. Принципы миссионной системы

1. **Kafka — единственная шина.** Никакого HTTP bridge. Все события — через Kafka
2. **MCP — единый протокол данных.** Никаких прямых API-вызовов
3. **Ранги — KPI агентов.** Без рангов нет измерения эффективности
4. **Автономия — не баг, а фича.** Миссия должна работать без связи неделями
5. **SKILL.md — единственный способ управления агентом.** Никакого жёсткого кода
6. **Findings — валюта миссии.** Каждый finding имеет ценность, влияет на ранги
