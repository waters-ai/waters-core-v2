# Бортовые журналы агентов WATERS

**Статус:** Доктрина. Фаза 1.
**Дата:** 2026-05-15 (ред. 2.0)

---

## 1. Назначение

Бортовой журнал — это хронологическая запись всех действий агента. Аналог судового журнала в морском деле: каждое событие фиксируется с меткой времени и тегом.

Журнал решает три задачи:
1. **Восстановление контекста** — при старте агент читает последние строки и понимает что делал
2. **Прозрачность** — любой другой агент (Архитектор, Конструктор) может прочитать журнал коллеги
3. **Аудит** — журнал — единственный источник правды о том что и когда делал агент

---

## 2. Формат записи

```
[ISO8601_TIMESTAMP] [TAG] описание
```

### Пример

```
[2026-05-15T10:00:00Z] [PLAN] Создать бортовой журнал Архитектора
[2026-05-15T10:00:00Z] [STEP] Изучен формат журнала Конструктора
[2026-05-15T10:00:00Z] [NOTE] Startup Sequence п.0 ссылается на constructor_journal.log — нужно исправить
[2026-05-15T10:00:00Z] [DONE] Бортовой журнал инициализирован
```

## 3. Теги

### Общие (для всех агентов)

| Тег | Значение | Когда использовать |
|-----|----------|-------------------|
| `[INIT]` | Инициализация | Создание журнала, первый запуск агента |
| `[STATE]` | Состояние | Текущее состояние системы/агента: RAM, серверы, статус |
| `[PLAN]` | План | Перед началом работы — что буду делать, пошагово |
| `[STEP]` | Шаг | Каждый выполненный шаг с результатом |
| `[NOTE]` | Замечание | Проблема, инсайт, найденная особенность |
| `[DONE]` | Завершено | Итог задачи: что сделано, статус |
| `[STOP]` | Остановка | Завершение сессии агента |

### Специфические для Архитектора

| Тег | Значение |
|-----|----------|
| `[VISION]` | Стратегическое видение, концепция |
| `[ANALYSIS]` | Анализ, сравнение, выводы |
| `[SPEC]` | Создание/изменение спецификации |
| `[ONTOLOGY]` | Работа с онтологией |
| `[ROADMAP]` | Работа с дорожной картой |
| `[DECISION]` | Принятое архитектурное решение |
| `[SYNC]` | Синхронизация с другими агентами |

### Специфические для Конструктора

| Тег | Значение |
|-----|----------|
| `[FIND]` | Найденный артефакт, файл, конфиг |
| `[STUDY]` | Изучение кода, документации |
| `[ACTION]` | Выполненное действие (деплой, билд, тест) |
| `[NEXT]` | Следующие шаги после текущей задачи |
| `[TEST]` | Результаты тестирования |
| `[AUDIT]` | Аудит кода/фронта/контракта |
| `[DEPLOY]` | Деплой на сервер |
| `[UPGRADE]` | Апгрейд системы |
| `[RULE]` | Новое правило или протокол |
| `[MISSION]` | Начало новой миссии |
| `[BUILD]` | Сборка проекта |
| `[FIX]` | Исправление бага |
| `[CHECK]` | Проверка (SPA routing, SSL, healthcheck) |
| `[CRED]` | Сохранение креденшелов |

## 4. Протокол ведения

### 4.1. Перед задачей — План

```
[2026-05-15T10:00:00Z] [PLAN] Описание задачи:
  1. Шаг первый
  2. Шаг второй
  3. Шаг третий
```

### 4.2. Во время — Шаги

```
[2026-05-15T10:00:00Z] [STEP] npm install — выполнено (15 сек)
[2026-05-15T10:00:00Z] [STEP] npm run build — 0 errors
[2026-05-15T10:00:00Z] [STEP] scp dist/ → сервер — готово
[2026-05-15T10:00:00Z] [NOTE] nginx не подхватил изменения — починил reload
```

### 4.3. После — Итог

```
[2026-05-15T10:00:00Z] [DONE] Фронтенд задеплоен. Статус: ✅
  Изменения: новый компонент WalletConnect
  Проблемы: bundle 1.4MB → 2.47MB (из-за Reown)
  Следующий шаг: проверить HTTPS
```

## 5. Правила

1. **Каждый чих — в журнал.** Любое действие, изменение, решение — записывается
2. **Одна строка — одно событие.** Не пиши длинные истории, разбивай на строки
3. **Время — ISO 8601 (UTC).** Всегда. Без часовых поясов
4. **Тег — в скобках, 4-8 символов.** `[PLAN]`, не `[ПЛАН]`, не `[Planing]`
5. **Пробел после тега обязателен.** `[STEP] текст`, не `[STEP]текст`
6. **При старте — читать последние 50 строк.** Восстановить контекст
7. **Новый файл — `[INIT]`.** При создании журнала первая запись — инициализация

## 7. Startup Sequence (чтение журнала при старте)

Каждый агент при запуске:
1. Читает **последние 50 строк** своего журнала
2. Восстанавливает контекст: что делал, на чём остановился
3. Если журнал пуст — записывает `[INIT]`
4. Записывает `[STATE]` с текущим состоянием системы

Пример для AGENTS.md Архитектора:

```
Startup Sequence:
  0. Read onboard log: прочитать infrastructure/logs/architect_journal.log (последние 50 строк)
  1. Connect MCP servers: filesystem, github, memory
  2. ...
```

## 9. Интеграция с Kafka

В будущем бортовые журналы могут дублироваться в Kafka topic `agent.journal.v1`:

```json
{
  "topic": "agent.journal.v1",
  "key": "architect",
  "value": {
    "agent": "agent.architect.v1",
    "timestamp": "2026-05-15T10:00:00Z",
    "tag": "DECISION",
    "message": "Форк TUI отменяется. Kafka-native Agent Runtime."
  }
}
```

Это позволит:
- Агрегировать логи всех агентов в одном месте
- Строить дашборд активности агентов
- Автоматический аудит решений
