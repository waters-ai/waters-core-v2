## 3.6 Восемь режимов DTN

Нода работает не в одном DTN-режиме, а в восьми различных, в зависимости от энергии, связи и ситуации.

```
            DTN-РЕЖИМЫ НОДЫ
               
     L0 ───── DTN-OFF      (связь есть, DTN не нужен)
     L1 ───── DTN-EVENT    (связь есть, но храним важное)
     L2 ───── DTN-STORE    (связи нет, копим всё)
     L2 ───── DTN-CHUNK    (большие данные → чанки)
     L2 ───── DTN-PRIORITY (сортировка очереди)
     L3 ───── DTN-SCHEDULE (синхронизация по часам)
     L3 ───── DTN-COMPRESS (максимальное сжатие)
     L4 ───── DTN-BEACON   (только пульс)
```

### 3.6.1 DTN-OFF (L0)

Связь стабильна, Kafka online. DTN выключен — все данные идут напрямую.

```
Поток: Агент → Kafka → MCC (реальное время)
Задержка: < 100 мс
Потребление: 0 Вт (DTN-модуль спит)
```

### 3.6.2 DTN-EVENT (L1)

Связь есть, но нода дублирует критичные данные в локальную DTN-очередь — на случай обрыва.

```
Триггер: любое finding с confidence > 0.9
Действие: 
  1. Отправить finding → Kafka (основной канал)
  2. Сохранить finding → offline queue (резерв)
  3. При обрыве → DTN-STORE автоматически

Потребление: +1 Вт (параллельная запись)
```

```rust
// dtn_event.rs
fn on_finding(finding: &Finding) {
    kafka_send(finding);                    // основной канал
    if finding.confidence > 0.9 {
        offline_queue_push(finding);        // резервная копия
    }
}
```

### 3.6.3 DTN-STORE (L2)

Связи нет. Нода копит все события в локальную очередь.

```
┌──────────┐    ┌──────────────┐    ┌──────────┐
│ Агент    │───▶│ offline_queue │───▶│ При flush │
│ Находка  │    │ .jsonl       │    │ → Kafka  │
└──────────┘    └──────────────┘    └──────────┘

Хранилище: /waters/offline/queue.jsonl
Формат: JSONL (одна строка = одно событие)
Лимит: 1000 событий или 50 MB
```

```json
// offline_queue.jsonl — пример записи
{"topic":"findings","key":"f_023","ts":"2026-05-16T18:23:00Z","retry":0}
{"topic":"telemetry","key":"t_4012","ts":"2026-05-16T18:23:01Z","retry":0}
{"topic":"heartbeat","key":"h_89","ts":"2026-05-16T18:24:00Z","retry":0}
```

### 3.6.4 DTN-CHUNK (L2 — большие данные)

Спектры, снимки, LiDAR-сканы — большие данные не помещаются в один DTN-пакет. Нода разбивает их на чанки.

```
Файл: spectrum_024.csv (12 MB)
        │
        ├── chunk_001 (4 MB) ──▶ Отправлен
        ├── chunk_002 (4 MB) ──▶ Отправлен  
        └── chunk_003 (4 MB) ──▶ Отправлен

Каждый чанк:
  - содержит seq (1/3, 2/3, 3/3)
  - checksum для проверки целостности
  - CRC32 заголовок
```

```rust
// dtn_chunk.rs
const CHUNK_MAX_SIZE: usize = 256 * 1024; // 256 KB на чанк

fn chunkify(data: &[u8]) -> Vec<Chunk> {
    data.chunks(CHUNK_MAX_SIZE).enumerate().map(|(i, chunk)| {
        Chunk {
            seq: i as u32,
            total: (data.len() + CHUNK_MAX_SIZE - 1) / CHUNK_MAX_SIZE,
            data: chunk.to_vec(),
            checksum: crc32(chunk),
        }
    }).collect()
}

fn dechunkify(chunks: &[Chunk]) -> Result<Vec<u8>> {
    let mut sorted = chunks.to_vec();
    sorted.sort_by_key(|c| c.seq);
    // Проверить: все seq от 0 до total-1
    // Проверить: checksum каждого чанка
    // Склеить
    Ok(sorted.iter().flat_map(|c| c.data.clone()).collect())
}
```

```
Процесс передачи:
  1. Нода А → Нода Б: CargoChunk (seq 0, total 3)
  2. Нода Б → Нода А: ACK (seq 0 received)
  3. Обрыв связи
  4. Связь восстановлена
  5. Нода А → Нода Б: "у тебя есть чанки?"
  6. Нода Б → Нода А: "есть seq 0, нет seq 1,2"
  7. Нода А → Нода Б: CargoChunk (seq 1, total 3)
  8. Нода А → Нода Б: CargoChunk (seq 2, total 3)
  9. Нода Б собирает файл
```

### 3.6.5 DTN-PRIORITY (L2 — сортировка очереди)

Когда связь появляется ненадолго, нода отправляет данные по приоритету:

```
Приоритет 1: findings (находки)          — отправляются первыми
Приоритет 2: orders.confirm (квитанции)  — вторыми
Приоритет 3: heartbeat (пульс)           — третьими
Приоритет 4: telemetry (телеметрия)      — четвёртыми
Приоритет 5: telemetry.raw (сырые данные) — последними, если хватит времени
```

```rust
fn flush_priority_queue() -> Vec<QueuedEvent> {
    let queue = read_offline_queue();
    let mut result = Vec::new();
    
    // priority 1: findings
    result.extend(queue.iter().filter(|e| e.topic == "findings"));
    // priority 2: confirmations
    result.extend(queue.iter().filter(|e| e.topic == "orders.confirm"));
    // priority 3: heartbeat (только последний)
    if let Some(hb) = queue.iter().rev().find(|e| e.topic == "heartbeat") {
        result.push(hb.clone());
    }
    // priority 4: telemetry
    result.extend(queue.iter().filter(|e| e.topic == "telemetry"));
    // priority 5: raw (только если есть место)
    let remaining = DTN_PACKET_MAX - result.len();
    result.extend(queue.iter()
        .filter(|e| e.topic == "telemetry.raw")
        .take(remaining));
    
    result
}
```

### 3.6.6 DTN-SCHEDULE (L3 — синхронизация по расписанию)

Нода не пробует отправить постоянно, а просыпается по расписанию:

```
Полигон Атакама:
  06:00 — восход, DTN-STORE → flush → DTN-OFF
  12:00 — пик солнца, проверка связи
  18:00 — закат, DTN-EVENT → DTN-STORE
  22:00 — DTN-SCHEDULE: принудительная отправка остатков
  00:00 — DTN-BEACON

Лаборатория Сантьяго (всегда онлайн):
  DTN-SCHEDULE не нужен — всегда DTN-OFF
```

```rust

```

### 3.6.7 DTN-COMPRESS (L3 — перед отправкой)

При низкой энергии нода сжимает данные перед отправкой — экономит и батарею (меньше передач), и время эфира.

```
┌──────────┐    gzip -6    ┌──────────┐
│ Исходные │ ────────────▶ │ Сжатые   │
│ 50 MB    │               │ ~8 MB    │
└──────────┘               └──────────┘

Алгоритмы:
  - text/JSON: gzip level 6 (сжатие 4-8×)
  - binary/spectra: bzip2 (сжатие 2-4×)
  - images: lossy JPEG, quality 85 (сжатие 10-20×)
  - LiDAR: laszip (сжатие 3-6×)

Время сжатия: ~5 сек на 50 MB
Выигрыш энергии: 6× меньше время передачи радиомодуля
```

### 3.6.8 DTN-BEACON (L4 — только пульс)

Минимальный режим. Нода только слушает и передаёт маяк.

```
Передача (каждые 5 минут):
  📡 Heartbeat: "node-1-atacama: L4, battery 12%"
  Размер: 64 байта
  Дальность: зависит от радио (LoRa: 5-15 км)

Приём:
  📡 Только аварийные команды: "return_to_base"
  Никаких других данных

Потребление: 0.1 Вт (радио + таймер)
```

## 3.7 Лазерные мосты (FSO) — прямая видимость между площадками

Когда между площадками есть прямая видимость, они соединяются лазером — Gbps, микросекунды задержки, без радиоэфира.

```
Полигон Атакама ───────── ЛАЗЕР ────────► Гора (ретранслятор)
  (лагерь)                                    │
        │                                     │
        │                          ЛАЗЕР      │
        │                                     ▼
        └────────────────────────────── Лаборатория Сантьяго
                                              (предгорье)

Скорость: 1-10 Gbps
Задержка: < 1 мс (лазер) vs 50-300 мс (Starlink)
Дальность: до 10 км (атмосфера), до 100 км (стратосфера)
```

### 3.7.1 Когда включается лазер

| Условие | Лазер | Starlink (радио) | DTN-очередь |
|---------|-------|-----------------|-------------|
| Ясно, день, есть линия | ✅ 10 Gbps | ❌ | ❌ |
| Облачно, есть линия | ❌ | ✅ 50 Mbps | ❌ |
| Ночь, есть линия | ✅ 10 Gbps | ✅ (резерв) | ❌ |
| Нет линии | ❌ | ✅ | ✅ (L2-L4) |
| вмешательство (песок/туман) | ❌ | ✅ 10 Mbps | ✅ |

### 3.7.2 Автомат выбора канала

```python
def select_channel(weather: str, los: bool, time: str) -> str:
    """Выбирает канал связи между площадками"""
    if los and weather == "clear":
        return "laser"       # 10 Gbps, 0.1 ms
    elif los and weather == "cloudy":
        return "starlink"    # 50 Mbps, 50 ms  
    elif not los and time == "day":
        return "starlink"
    elif not los:
        return "dtn"         # offline queue
    else:
        return "starlink"    # fallback
```

### 3.7.4 Дрон как лазерный ретранслятор

Когда дрон в воздухе на высоте, он может быть лазерным мостом между площадками:

```
Дрон (H=400 м)
  ├── ЛАЗЕР ◄── Лагерь (на земле)
  └── ЛАЗЕР ──► Сантьяго (через гору, 40 км)

Пропускная способность: 2.5 Gbps (адаптация под вибрации дрона)
Задержка: 0.3 мс
```

```rust
// drone_laser_bridge.rs
pub struct DroneLaserBridge {
    pub drone_id: String,
    pub altitude_m: f64,
    pub ground_station: String,
    pub target_station: String,
    pub bandwidth_gbps: f64,
    pub lock_status: bool,      // захват цели
    pub vibration_comp: bool,   // компенсация вибраций
}

impl DroneLaserBridge {
    pub fn is_line_of_sight(&self, weather: &Weather) -> bool {
        weather.visibility_km > 20.0 && !weather.is_dust_storm()
    }
    
    pub fn estimated_bandwidth(&self, weather: &Weather) -> f64 {
        match weather.condition {
            "clear" => 10.0,
            "light_haze" => 5.0,
            "moderate_haze" => 2.5,
            "dust" => 0.0,   // лазер не работает в пыли
            _ => 1.0,
        }
    }
}
```

### 3.7.5 Лазер vs радио vs DTN — когда что

| Характеристика | Лазер (FSO) | Starlink | DTN |
|---------------|-------------|----------|-----|
| Скорость | 1-10 Gbps | 50-220 Mbps | 0 (офлайн) |
| Задержка | 0.1 ms | 20-50 ms | Часы |
| Зависимость от погоды | Высокая | Средняя | Нет |
| Прямая видимость | Обязательна | Не нужна | Не нужна |
| Энергопотребление | 50 Вт | 30 Вт | 5 Вт |
| Что передавать | Спектры, LiDAR, видео | Телеметрия, heartbeat | Только findings |
| Когда использовать | L0 (макс мощность) | L0-L1 | L2-L4 |

### 3.7.6 Переключение лазер ↔ Starlink без потери данных

```python
def switch_laser_to_starlink():
    """Плавное переключение с лазера на Starlink при ухудшении погоды"""
    # 1. Включить Starlink параллельно
    starlink.on()
    
    # 2. Дождаться стабильного соединения
    while not starlink.is_stable():
        sleep(1)
    
    # 3. Переключить трафик на Starlink
    route.traffic_to("starlink")
    
    # 4. Выключить лазер
    laser.off()
    
    # 5. Оповестить Kafka
    kafka.publish("power.v1", {
        "channel": "starlink",
        "reason": "weather_degraded",
        "laser_uptime_sec": laser.uptime_seconds()
    })
```

### 4.6 Как группы делят общие задачи

```
MCC ──orders.v1──► ГРУППА ──► Координатор группы
```

**Принцип:** MCC ставит задачу **группе**, а группа сама решает, какой агент что делает. Координатор группы (самый опытный агент) распределяет подзадачи.

```rust
// Как группа принимает задачу
impl Group {
    fn on_order(&mut self, order: &Order) -> Vec<SubTask> {
        match order.type {
            "search" => self.coordinator.delegate_search(order),
            "analyze" => self.coordinator.delegate_analysis(order),
            "report" => self.coordinator.delegate_report(order),
            _ => vec![SubTask::single(order)], // одному агенту
        }
    }
}

impl Coordinator {
    fn delegate_search(&self, order: &Order) -> Vec<SubTask> {
        vec![
            SubTask::new(&self.navigator, "построить маршрут"),
            SubTask::new(&self.drone_operator, "облететь сектор"),
            SubTask::new(&self.geologist, "анализировать находки"),
        ]
    }
}
```

### 4.9 Шесть нод — одна задача — чужие лаборатории через мосты

Главная сила waters-node: 6 агентов в 6 концах мира решают одну задачу, используя оборудование друг друга через MCP-мосты.

```
🔬 Лаборатория Токио (LIBS-спектрометр) ──mcp──┐
🛰️ Спутниковая станция Дубай (SAR) ─────mcp──┤
🏔️ Полевой лагерь Анды (образцы) ──────mcp──┤── ОДНА ЗАДАЧА
🧪 Лаборатория Берлин (масс-спектрометр) ─mcp──┤   "найти метеорит"
🏛️ Музей Лондон (эталонные образцы) ────mcp──┤
💻 MCC Москва (координация) ──────────────mcp──┘

У каждого своё оборудование. Все работают через bridges.
```

| Нода | Город | LLM | Местное оборудование | Мост (bridge) |
|------|-------|-----|---------------------|---------------|
| 1 | **Анды** (поле) | DeepSeek Flash | Дроны, LIBS-полевой, GPS | `mcp-drone`, `mcp-libs-field` |
| 2 | **Токио** | Claude | LIBS-лабораторный, SEM, XRD | `mcp-libs-lab`, `mcp-sem`, `mcp-xrd` |
| 3 | **Дубай** | GPT-4o | Спутниковая станция, SAR | `mcp-sar`, `mcp-satellite` |
| 4 | **Берлин** | Ollama | Масс-спектрометр, ICP-MS | `mcp-icp-ms`, `mcp-mass-spec` |
| 5 | **Лондон** | DeepSeek Pro | Музейная коллекция, эталоны | `mcp-meteorite-collection` |
| 6 | **Москва** | OpenRouter | Координация, Kafka, MCC | `mcp-kafka`, `mcp-coordinator` |

#### Как это работает

```
Задача: "метеорит f_023 — определить состав"

Шаг 1: Поле (Анды) находит образец
  → дрон → LIBS-полевой → "похож на хондрит"
  → finding: "f_023, conf 0.6"
  → Kafka → все ноды видят

Шаг 2: Токио берёт спектр через свой LIBS
  → Агент Токио: "дайте доступ к вашему LIBS"
  → Но у Токио СВОЙ LIBS (лабораторный, точнее)
  → bridge: mcp-libs-lab
  → спектр: "L6 ordinary chondrite, conf 0.95"
  → finding: уточнение → Kafka

Шаг 3: Берлин проверяет состав через масс-спектрометр
  → bridge: mcp-icp-ms  
  → изотопы: "Fe 27%, Ni 1.5%, conf 0.99"
  → finding: подтверждение → Kafka

Шаг 4: Лондон сверяет с эталоном
  → bridge: mcp-meteorite-collection
  → "совпадает с эталоном L6 #4421"
  → finding: финальное подтверждение → Kafka

Шаг 5: Москва собирает всё в отчёт
  → Kafka findings → coordinator → reporter
  → отчёт → MCC → Архитектор
```

#### Ключевой принцип

```bash
# Агент в Токио использует спектрометр в Токио
# Агент в Берлине использует масс-спектрометр в Берлине
# Агент в Лондоне использует коллекцию в Лондоне
# Все работают на ОДНУ задачу

# bridges.json каждой ноды содержит только её оборудование:
# node-tokyo/bridges.json → ["mcp-libs-lab", "mcp-sem", "mcp-xrd"]
# node-berlin/bridges.json → ["mcp-icp-ms", "mcp-mass-spec"]
# node-london/bridges.json → ["mcp-meteorite-collection"]

# Никто никуда не едет. Оборудование не перемещается.
# Данные перемещаются. Через Kafka. Через bridges.
# 6 нод = 6 лабораторий = 1 задача.
```

#### Цепочка компетенций

```
Поле (сбор) ──► Токио (состав) ──► Берлин (изотопы) ──► Лондон (эталон) ──► Москва (отчёт)
     │               │                  │                    │                 │
     │     mcp-libs-lab     mcp-icp-ms    mcp-meteorite    mcp-kafka
     │               │                  │    collection        │
     ▼               ▼                  ▼         ▼           ▼
   Kafka ◄───────── Kafka ◄─────────── Kafka ◄── Kafka ◄──── Kafka
```

## 5. Автомат переключения режимов (DTN + питание + связь)

```
                    DTN-OFF (L0)
                       │
              связь потеряна
                       │
                       ▼
                  DTN-EVENT (L1)
                  ┌────┴────┐
                  │         │
          связь    │   10 мин нет
        вернулась  │    связи
                  │         │
                  ▼         ▼
              DTN-OFF   DTN-STORE (L2)
                          │
              ┌───────────┼───────────┐
              │           │           │
         большие     battery    battery
         данные      < 50%     < 20%
              │           │           │
              ▼           ▼           ▼
         DTN-CHUNK   DTN-SCHEDULE  DTN-COMPRESS
              │           │           │
              └───────────┼───────────┘
                          │
                     battery < 5%
                          │
                          ▼
                    DTN-BEACON (L4)
```

### Пример работы автомата

```
14:00 — DTN-OFF: связь есть, всё онлайн
14:05 — обрыв Starlink
14:05 → DTN-EVENT: дублируем критичные данные
14:15 → DTN-STORE: связь не вернулась, копим всё
14:20 — спектрометр выдал 50 MB данных
14:20 → DTN-CHUNK: разбиваем на чанки по 256 KB
14:30 — battery упала до 40%
14:30 → DTN-PRIORITY + DTN-SCHEDULE: 
         отправляем только findings, по расписанию
15:00 — battery 18%
15:00 → DTN-COMPRESS: сжимаем всё перед отправкой
15:30 — battery 4%
15:30 → DTN-BEACON: только heartbeat
16:00 — связь восстановлена
16:00 → DTN-OFF: offline_queue → flush → всё в Kafka
```

## 8. Аварийные протоколы

| Ситуация | Действие | Kafka |
|----------|---------|-------|
| Потеря связи с Атакамой > 5 мин | Дрон возвращается на базу, нода — L4 | offline queue |
| Солнечная панель не заряжает | Дизель-генератор автозапуск | power.v1 alert |
| LLM недоступен | Fallback на Ollama local (нода 1) | alerts.v1 |
| Kafka брокер недоступен | Все ноды — DTN-режим | heartbeat.v1 offline |
| Найден метеорит > 1 кг | Приоритетный слот в Kafka | orders.v1 urgent |
