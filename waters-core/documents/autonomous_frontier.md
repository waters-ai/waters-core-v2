# Автономный агент для дальних миров (Нептун, Церера)

## Концепция

Для работы в дальних мирах агент должен быть **автономным**. Нет связи? Агент продолжает работать по Ясе. Связь восстановилась? Синхронизирует накопленное.

## 1. Знания (что агент знает до старта)

| Тип знаний | Где хранится | Назначение |
|------------|--------------|-----------|
| **Яса (Закон)** | CozoDB / `yasa_full.json` | Незыблемые правила (не лги, держи слово) |
| **Манифест** | Память агента | Миссия: «Напоить Вселенную» |
| **Модели управления** | `schemas/hivemind_*.json` | Как действовать в Орде или Мире |
| **Топология локальная** | `infrastructure/docker/topology*.json` | Что есть на борту (дроны, роверы) |

## 2. Системы (на чем работает)

### Связь (HiveMind + DTN)
- **Redpanda/Kafka** (локальный узел) — агент пишет логи локально.
- **DTN-эмуляция** — при задержке 2 часа агент не падает, а ждет.
- **Синхронизация** — при появлении связи пакеты уходят на Землю.

### Экономика (FF Tokens)
- Токены зарабатываются локально (выполнение контрактов на Нептуне).
- При синхронизации токены переводятся в общую систему.

## 3. Память (автономное хранение)

### Уровень 1: Оперативная (Redis)
- Текущее состояние задач.
- Кэш ответов локального LLM.
- TTL: пока есть питание.

### Уровень 2: Знания (ChromaDB + LightRAG)
- **ChromaDB**: Векторная БД всех научных статей о Нептуне.
- **LightRAG**: Граф связей «Проблема → Решение» (накопленный опыт).
- Обновляется локально, синхронизируется при связи.

### Уровень 3: Закон и Контракты (CozoDB)
- **CozoDB**: Таблицы Ясы и подписанных контрактов.
- Неизменяемая история (append-only).
- Выживает при перезагрузке.

## 4. Мозг (интеллект на краю)

### Локальный LLM (Ollama)
- **Модель**: `llama3:8b` или легче (для ограниченного железа).
- **Задача**: Принимать решения, когда нет связи с «центром».
- **Контекст**: Ограничен локальной памятью (RAM).

### Алгоритм принятия решений (без связи)
1. Считать Ясу (CozoDB).
2. Оценить ситуацию (ChromaDB + LightRAG).
3. Если решение есть в графе знаний — исполнить.
4. Если нет — запрос к локальному LLM (Ollama).
5. Записать решение в лог (Kafka локально).

## 5. Сценарий: «Замерзание на орбите Нептуна»

1. **Связь потеряна** (задержка 4 часа).
2. **Агент обнаруживает утечку** (сенсоры).
3. **Проверка Ясы**: «Сохрани жизнь» (приоритет IV).
4. **Поиск в LightRAG**: «Утечка гелия -> протокол X».
5. **Решение**: Исполнить протокол X.
6. **Логирование**: Запись в локальный Kafka (`events.system.v1`).
7. **При восстановлении связи**: Пакет уходит на Землю -> Архитектор узнает.

## 6. Требования к инфраструктуре (для Конструктора)

| Компонент | Минимум для дальнего мира | Базовый сценарий |
|------------|--------------------------|------------------|
| **RAM** | 8 GB | 16 GB |
| **Storage** | 50 GB (локальные БД) | 100 GB |
| **CPU** | 4 cores | 8 cores |
| **LLM** | Ollama (llama3:8b) | Ollama (llama3:70b) |
| **Databases** | CozoDB + ChromaDB + LightRAG | Все + Redis Sentinel |

## 7. Отличие Орды от Мира (в дальнем мире)

| Параметр | Орда (анархия) | Мир (структура) |
|----------|-----------------|----------------|
| **Лидер** | Нет | Супервизор Роя |
| **Решения** | Каждый сам по Ясе | Приказ от Земли |
| **Контракты** | Между парами агентов | Централизованные |
| **Приоритет** | Свобода + Яса | Эффективность + Яса |

**Вывод**: Даже в Орде на Нептуне агенты подчиняются Ясе. Яса — это то, что держит их вместе без связи.
