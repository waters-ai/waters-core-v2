# Протокол кэширования Redis — WATERS

## Назначение

Redis используется как слой кэширования для уменьшения количества запросов к внешним AI (экономия токенов) и ускорения доступа к часто запрашиваемым данным.

## 1. Какие запросы кэшировать

### Кэшируемые запросы

| Тип запроса | Пример | Причина |
|-------------|--------|---------|
| Частые запросы к AI | Повторяющиеся вопросы к Claude/GPT | Экономия токенов |
| Результаты поиска | Поиск по одному и тому же запросу | Ускорение ответа |
| Конфигурации компонентов | Настройки Docker, Kafka, Redis | Стабильные данные |
| Схемы JSON | HiveMind схемы, схемы событий | Редко меняются |

### Не кэшируемые запросы

- Запросы с временным контекстом (текущая дата/время)
- Уникальные запросы (вероятность повтора < 5%)
- Запросы с чувствительными данными

## 2. Структура ключей

### Формат ключей

```
cache:<source>:<hash>
```

### Примеры

| Ключ | Описание |
|------|----------|
| `cache:ai:abc123def456` | Ответ от внешнего AI |
| `cache:chromadb:xyz789` | Результат запроса к ChromaDB |
| `cache:search:exa:qwerty` | Результат поиска через Exa |
| `cache:schema:hivemind_military` | Схема HiveMind (военная) |

### Генерация hash

Hash генерируется как SHA-256 от нормализованного запроса:

```python
import hashlib
import json

def generate_cache_key(source, query):
    normalized = json.dumps(query, sort_keys=True)
    hash_value = hashlib.sha256(normalized.encode()).hexdigest()[:16]
    return f"cache:{source}:{hash_value}"
```

## 3. TTL (время жизни)

### Политика TTL

| Тип данных | TTL | Обоснование |
|------------|-----|-------------|
| Частые запросы к AI | 1 час (3600s) | Высокая вероятность повтора |
| Результаты поиска | 6 часов (21600s) | Средняя стабильность |
| Конфигурации | 24 часа (86400s) | Редко меняются |
| Схемы JSON | 7 дней (604800s) | Стабильные данные |
| Редкие запросы | 24 часа (86400s) | Защита от устаревания |

### Установка TTL

```bash
# При сохранении в кэш
SETEX cache:ai:abc123 3600 '{"response": "..."}'

# Проверка TTL
TTL cache:ai:abc123
```

## 4. Как проверять кэш перед запросом к AI

### Алгоритм

1. Сформировать нормализованный запрос
2. Сгенерировать cache key
3. Проверить наличие в Redis: `GET cache:ai:<hash>`
4. Если есть — вернуть из кэша (экономия токенов)
5. Если нет — выполнить запрос к AI, сохранить в кэш

### Псевдокод

```python
def get_ai_response(query):
    # Генерируем ключ
    cache_key = generate_cache_key("ai", query)

    # Проверяем кэш
    cached = redis.get(cache_key)
    if cached:
        log("Cache hit for query")
        return json.loads(cached)

    # Кэш пуст — запрашиваем AI
    response = call_external_ai(query)

    # Сохраняем в кэш (TTL 1 час)
    redis.setex(cache_key, 3600, json.dumps(response))

    return response
```

## 5. Инвалидация кэша

### Сценарии инвалидации

| Сценарий | Действие |
|----------|----------|
| Обновление схемы | `DEL cache:schema:*` |
| Обновление конфигурации | `DEL cache:config:*` |
| Принудительная очистка | `FLUSHDB` (только для dev) |
| По тегу | Использовать RedisJSON или отдельную индексную структуру |

### Команды инвалидации

```bash
# Удалить конкретный ключ
DEL cache:ai:abc123

# Удалить по паттерну (осторожно в проде!)
redis-cli --scan --pattern "cache:schema:*" | xargs redis-cli DEL
```

## Интеграция с Нервной системой

- Кэш метрик отправляется в `metrics.raw.v1` (cache_hit, cache_miss)
- События инвалидации в `events.system.v1`
- Конфигурация кэша хранится в CozoDB (таблица `cache_rules`)

## Мониторинг

| Метрика | Описание | Цель |
|---------|----------|------|
| cache_hit_rate | Доля попаданий в кэш | > 60% |
| cache_miss_rate | Доля промахов | < 40% |
| tokens_saved | Сэкономленные токены | Максимизация |
| eviction_count | Количество вытеснений | Минимизация |
