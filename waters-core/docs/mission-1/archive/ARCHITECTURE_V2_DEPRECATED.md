## 2. Принципиальные изменения после гибели 237

| Было (237) | Стало (2026-05-15) |
|-----------|-------------------|
| Ollama на 237, SSH-туннели | Ollama на 238 (ХАБ) + Ollama на миссионном сервере |
| Neo4j на 237 | LightRAG на 238 (Neo4j исключён) |
| TimescaleDB на 237 | Очереди в Redpanda на 238 |
| Три сервера: 237 + 238 + 167 | Два сервера: 238 + новый миссионный |
| N/A | DTN-эмуляция на миссионном сервере (tc-netem) |

## 3. Протоколы обмена

### 3.1. ХАБ (238) → Миссионный сервер (orders)

```
Kafka: mission.1.orders.v1
Формат: JSON-RPC 2.0
{
  "jsonrpc": "2.0",
  "method": "mission.order",
  "params": {
    "type": "search_area|recalibrate|abort|pause",
    "payload": { "...": "..." },
    "ttl": 3600
  }
}
```

Доставка: MCP-адаптер mcp-bridge на миссионном сервере читает HTTP endpoint на 238.
Если связи нет — mission-control работает в L2 (автономном) режиме.

### 3.2. Миссионный сервер → ХАБ (findings)

```
HTTP POST https://238.hub.waters/mission/1/findings
{
  "mission_id": "mission-1",
  "findings": [
    {
      "id": "uuid",
      "type": "meteorite_candidate|anomaly|status",
      "confidence": 0.87,
      "data": { "...": "..." },
      "timestamp": "2026-05-15T12:00:00Z"
    }
  ],
  "heartbeat": {
    "uptime": 3600,
    "agents_active": 12,
    "memory_usage": 0.6
  }
}
```

### 3.3. Внутренний протокол миссионного сервера (Redis pub/sub)

```
Каналы Redis:
  agent:heartbeat    — healthcheck агентов
  mission:state     — состояние миссии (collecting|analyzing|idle|error)
  mission:findings  — новые находки
  mission:orders    — внутренние приказы RLM Pro
```

Формат — JSON, все сообщения пишутся в RocksDB для durability.

## 4. Уровни автономии миссионного сервера

| Уровень | Связь с 238 | LLM | Поведение |
|---------|-------------|-----|-----------|
| **L0** | ✅ | DeepSeek API | Полный центр, все возможности |
| **L1** | ✅ | Ollama локально | Экономия API-ключей |
| **L2** | ❌ | Ollama локально | **Автономный**: очередь findings, sync при восстановлении |
| **L3** | ❌ | Ollama урезанная | **Энергосбережение**: только критичные агенты |
| **L4** | ❌ + нет энергии | — | **Safe mode**: только логирование |

Реализуется в RLM Pro через модуль `mission_mode.rs`:
- heartbeat к 238 раз в 60 секунд (L0-L1)
- При потере связи — переход на L2 за 30 секунд
- При восстановлении — batch-отправка findings через HTTP bridge

## 5. Memory стек

### На ХАБ (238):
```
Kafka       — глобальная шина событий (20+ топиков)
Redis       — кэш, pub/sub, глобальное состояние
ChromaDB    — векторная память (все findings всех миссий)
LightRAG    — граф знаний (связи между метеоритами, траекториями)
```

### На миссионном сервере:
```
Redis       — состояние миссии, pub/sub (недолговечное)
RocksDB     — durable очередь задач (встроена в TUI, переживает рестарт)
```
ChromaDB и LightRAG на миссионном сервере НЕ СТАВИМ.
Только на 238 — findings отправляются туда через bridge.

**Обоснование:** ChromaDB + LightRAG требуют ~2GB RAM и регулярного CPU, что на миссионном сервере роскошь. Все вектора и графы строятся централизованно на 238.

## 6. DTN-эмуляция

На миссионном сервере через tc-netem:

```bash
# Эмуляция задержки Земля-Луна: 1.3 секунды
sudo tc qdisc add dev eth0 root netem delay 650ms 50ms distribution normal

# Эмуляция потери пакетов (для тестов)
sudo tc qdisc change dev eth0 root netem delay 650ms 50ms loss 0.5%

# Удаление эмуляции
sudo tc qdisc del dev eth0 root
```

В mission-control модуль `dtn.rs`:
- Читает latency_plan.json
- Применяет tc-netem правила при старте
- Логирует latency в Redis (mission:metrics:latency)

## 7. Ключевые решения по архитектуре

1. **Один тип Flash-агента** — не 4 Rust-структуры, а 1 универсальный FlashAgent с разными SKILL.md
2. **Никакого Neo4j** — LightRAG достаточно для графовых связей (на 238)
3. **Никакой TimescaleDB** — метрики в Kafka с retention 7 дней
4. **RocksDB — единственная постоянная память на миссионном сервере** (Redis — только для состояния)
5. **MCP как единственный протокол** — все внешние источники (NASA, спектры, траектории) через MCP
6. **DTN — на уровне ядра** — встроен в mission-control, а не внешний сервис
