## 1. Резюме

### Проблема

Существующая архитектура waters-node (v0.5.0) полностью полагается на LLM. Каждый запрос агента → LLM → ответ. Это даёт:
- 100% зависимость от внешнего API (DeepSeek/Claude)
- Галлюцинации: LLM может выдать ложный ответ без механизма проверки
- Нет накопления знаний: агенты не помнят, что они делали и чему научились
- Нет A2A-знаний: агенты общаются текстом, а не структурированными ситуациями

### Решение: CognitionLayer

Слой познания между агентами и LLM, который:
- Перехватывает **80% запросов** на уровне ассоциативной памяти (ASP), без вызова LLM
- Верифицирует каждый ответ LLM через **Бонгард-фильтр** перед записью в память
- Накопляет знания как **ситуации** (контекст + решение), а не как текст
- Распространяет знания между нодами через **prion-механизм**
- Устраивает **геномный взрыв** при появлении DeepSeek: за 15 мин синк всех 6 нод
- Экспериментально определяет **лучшую конфигурацию алгоритмов** через 6 типов нод

### Ценность

| Сценарий | Без CognitionLayer | С CognitionLayer |
|----------|------------------|-----------------|
| Офлайн-режим | L4 (safe mode), SOS | ASP + Edge LLM, 80% задач решается |
| Стоимость токенов | 100% запросов → DeepSeek | 2% запросов → DeepSeek |
| Галлюцинации | LLM говорит что угодно | Бонгард проверяет каждое утверждение |
| Обучение агента | Ноль. Агент не умнеет | Каждый ответ → ситуация → ASP → прион |
| 6 нод вместе | Каждая нода учится отдельно | Геномный взрыв: 15 мин = вся сеть знает |

### 2.2 Структура ситуации (ядро данных)

```rust
pub struct Situation {
    pub id: u64,
    pub domain: String,              // domain: "code", "security", "access"
    pub context_hash: u64,           // blake3 хэш контекста
    pub scenario: Scenario,          // что произошло
    pub action: Action,              // что сделали
    pub solution: String,            // код / решение
    pub confidence: f64,             // Байесовская: 0.0 - 1.0
    pub verifications: u32,          // сколько раз подтверждено
    pub timestamp: DateTime<Utc>,
    pub source_node: NodeId,         // какая нода создала
    pub source_agent: AgentId,       // какой агент
    pub weight: f64,                 // синаптический вес (Baldwin)
    pub prion_count: u32,            // сколько раз распространена
    pub traversed_nodes: Vec<NodeId>, // через какие ноды прошла
    pub code_validated: bool,        // прошла ли code review
    pub security_checked: bool,      // прошла ли security check
}
```

### 2.3 Пример работы CognitionLayer

```
Агент implementer получил задачу: "написать функцию парсинга JSON"

ШАГ 1 — ASP запрос:
  cognize("написать функцию парсинга JSON", "code", context)
  → SDM: ищу похожие ситуации...
  → Situator: контекст → "реализация парсера на Rust"
  → ZHURAVLEV: классификация по признакам → "serialization"
  
  HIT (80% случаев):
    → ситуация{id: 42, solution: "serde::Deserialize + custom error", confidence: 0.94}
    → агент получает готовый паттерн
    → LLM не вызывалась, 1ms, 0 токенов

  MISS (20% случаев):
    → LLM (Edge или DeepSeek) генерирует ответ
    → Ответ проходит:
       L1 Bongard: TRUE (код компилируется, логика верна)
       L2 Federator: +weight (совпало с лучшей практикой)
       → Situator сжимает в ситуацию
       → ASP сохраняет
       → Prion соседям
       → В следующий раз: HIT
```

## 3. Агенты ноды (11 встроенных)

### 3.1 Состав

```
┌─────────────────────────────────────────────────────────┐
│  7 TUI-АГЕНТОВ (существующие, из TUI)                   │
│                                                          │
│  general     — общие запросы, диалог                     │
│  explore     — исследование кода, поиск                  │
│  plan        — планирование перед задачей                │
│  review      — ревью кода                               │
│  implementer — пишет код                                │
│  verifier    — тестирует, проверяет                     │
│  custom      — пользовательские сценарии                │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  4 НОВЫХ АГЕНТА НОДЫ                                    │
│                                                          │
│  access_open   — решает: открыть доступ                  │
│  access_close  — решает: закрыть доступ                  │
│  security      — следит за нодой, Яса                    │
│  task_manager  — назначает задачи, трекинг              │
└─────────────────────────────────────────────────────────┘
```

### 3.2 Механизм Access Open/Close (два ключа)

```
Внешний запрос на доступ к ноде
  → access_open проверяет: контекст, identity, намерение
  → access_close проверяет: риски, ресурсы, Яса
  │
  ├── СОГЛАСНЫ (оба Approve) → доступ открыт/закрыт
  ├── НЕ СОГЛАСНЫ (+ / -) → эскалация пользователю
  └── ОБА ОТКЛОНИЛИ → блокировка

Каждое решение → ситуация в ASP:
  "ситуация: запрос доступа от {id}, решение: {approve/reject}, контекст: {context}"
  → confidence растёт с каждым подтверждением пользователя
  → через N циклов агенты согласуются без пользователя
```

### 3.3 Security Agent

```
Фоновый мониторинг:
  - YASA-проверка каждого входящего/исходящего действия
  - Анализ логов: аномалии, подозрительные паттерны
  - Проверка кода перед deploy (в паре с code_review_pipeline)
  - Quorum-запрос соседям при подозрительной ситуации
  - Прерывание задачи при нарушении Ясы

Знания security → в ASP:
  "ситуация: паттерн атаки X, реакция: Y, сработало: да/нет"
  → все ноды учатся на атаках друг друга
```

### 3.4 Task Manager

```
Приём задачи:
  1. Парсинг: domain, complexity, required skills
  2. ASP lookup: "была ли похожая задача?"
     → HIT: готовый план выполнения, estimated time
     → MISS: LLM генерирует план
  3. Назначение: выбрать агента (implementer / explore / custom)
  4. Трекинг: статус, время, результат
  5. Результат → ситуация → ASP → прион
```

## 4. 6 типов нод — экспериментальная архитектура

### 4.1 Конфигурации

```
НОДА 1 — SDM + Situator (чистая ассоциативная память)
  Слои: ASP(SDM + Situator)
  Вопрос: хватит ли чистой памяти для 80% hit?
  Гипотеза: нет, нужна верификация

НОДА 2 — SDM + Situator + Bongard
  Слои: ASP + Bongard
  Вопрос: насколько Бонгард снижает процент ложных ситуаций?
  Гипотеза: confidence вырастет на 15-20%

НОДА 3 — SDM + Situator + Bongard + Federator
  Слои: ASP + Bongard + Federator
  Вопрос: ускоряет ли дофамин-обратная связь обучение?
  Гипотеза: да, но утяжеляет систему

НОДА 4 — SDM + Situator + Bongard + Federator + Edge LLM
  Слои: ASP + Bongard + Federator + Edge(1.5B)
  Вопрос: даёт ли Edge LLM прирост качества кода?
  Гипотеза: +15% к решаемости задач в офлайне

НОДА 5 — SDM + Situator + Bongard + Federator + Prion (усиленный)
  Слои: ASP + Bongard + Federator + Prion(stigmergy + quorum)
  Вопрос: быстрее ли учится сеть с активными прионами?
  Гипотеза: да, но возможен шум

НОДА 6 — SDM + Glushkov + MGUA + Tensor (мини-Центр Познаний)
  Слои: ASP + Glushkov recursion + MGUA Ivakhnenko + Osledets tensor
  Вопрос: даёт ли математическое сжатие преимущество?
  Гипотеза: геном компактнее, но latency выше
```

### 4.2 Метрики сравнения

| Метрика | Описание | Цель |
|---------|----------|------|
| ASP hit rate | % ответов из ASP без LLM | > 80% |
| Confidence | средняя уверенность | > 0.85 |
| False positive | % ложных ситуаций | < 1% |
| Task completion | % решённых без DeepSeek | > 90% |
| Genome density | ситуаций на MB | max |
| Code quality | % кода, прошедшего review | > 85% |
| Convergence | циклов до стабильности | min |

## 5. Геномный взрыв (Genome Burst)

### 5.1 Цикл

```
ФАЗА A — Офлайн-накопление (часы/дни):
  Каждая нода собирает ситуации в genome_queue.
  ASP + Edge LLM работают. P2P-прионы работают.
  Задачи требующие DeepSeek → task:pending.
  Ситуации с confidence < порога → ждут верификации.

ФАЗА B — Quorum Trigger:
  DeepSeek появился (или хозяин подключил).
  Quorum-сигнал: "DeepSeek live, window 15 min".
  Каждая нода:
    - замораживает ASP-запись
    - пакует genome:
        situations: Vec<CompressedSituation>,  // тензоры Оселедца
        pending_tasks: Vec<Task>,               // что спросить
        confidence_map: HashMap<u64, f64>,      // текущие confidence
        failed_verifications: Vec<u64>,          // что Бонгард отклонил
        domain_stats: HashMap<String, u32>,

ФАЗА C — Синк-окно (2 мин из 15):
  Все 6 нод сливают геномы координатору (Центр Познаний / лидер-нода).
  Координатор:
    - дедуплицирует (Zhuravlev: algebra recogn)
    - классифицирует (Pospelov: признаковый индекс)
    - строит единый геном сети (Glushkov: recursive compression)
    - confidence пересчитывает (Bayes: по 6 выборкам)

ФАЗА D — DeepSeek-сессия (10 мин из 15):
  Только сложные задачи из task:pending.
  DeepSeek отвечает → Situator сжал → Bongard проверил → в единый геном.
  DeepSeek верифицирует ситуации с низким confidence.
  DeepSeek ревьюит код, написанный Edge LLM.

ФАЗА E — Компиляция и рассылка (3 мин из 15):
  Единый геном → Glushkov recursion → Osledets tensor compression.
  Baldwin-фиксация: успешные → ядро, неудачные → испарение.
  Рассылка всем 6 нодам.
  Каждая нода распаковывает геном в свою ASP.

ФАЗА F — DeepSeek отключился:
  Ноды снова в офлайне.
  ASP всех 6 нод обновлены. ASP hit rate вырос.
```

### 5.2 Ключевой эффект

```
Без геномного взрыва:
  Каждая нода учится отдельно.
  6 нод = 6× повторных LLM-запросов.

С геномным взрывом:
  Все ситуации собраны → дедуплицированы → сжаты.
  70% overlap между нодами → выкинут.
  DeepSeek отвечает только на 0.3% уникальных задач.
  Ответ → сразу во все 6 ASP.
  Экономия: 20× меньше вызовов LLM.
```

## 6. Протокол A2A (Agent-to-Agent Knowledge)

### 6.1 Формат сообщения

```rust
pub struct KnowledgeMessage {
    pub msg_type: MsgType,          // situation | verification | prion | quorum
    pub situation_id: u64,
    pub domain: String,
    pub context_hash: u64,
    pub scenario: Scenario,
    pub solution: String,
    pub confidence: f64,
    pub source_node: NodeId,
    pub source_agent: AgentId,
    pub traversed_nodes: Vec<NodeId>,
    pub timestamp: DateTime<Utc>,
}

pub enum MsgType {
    Situation,         // новая ситуация
    Verification,      // результат верификации
    Prion,             // prion-распространение
    QuorumRequest,     // запрос подтверждения
    QuorumResponse,    // ответ на quorum
}
```

### 6.2 Пример A2A-диалога

```
Агент1(implementer) → Агент2(verifier):
  Message{type: Situation, id: 42, domain: "code",
          scenario: "parse JSON", solution: "serde::Deserialize",
          confidence: 0.91, source_node: "n1"}

Агент2(verifier) → Агент1:
  Message{type: Verification, id: 42, result: TRUE,
          confidence: 0.94, note: "тесты пройдены"}

Агент2 → Агент3(review):
  Message{type: Prion, id: 42, ...}
  // misfold: Агент3 теперь тоже знает ситуацию 42
```

## 7. Память и кэши

### 7.1 Трёхуровневая система памяти

```
L0: ASP (локально на ноде) — 500MB, 1ms
  Технология: SDM Kanerva + Situator Pospelov
  Хранит: ситуации-сценарии, confidence, weight
  Eviction: LRU + синаптический вес (частые не вытесняются)

L1: Redis shared (6 нод, лидер) — 500MB, 5ms
  Разделы:
    - llm:cache — кэш ответов LLM (ключ: blake3(query))
    - asp:shared — общие ситуации (prion-буфер)
    - task:state — состояние общей задачи
    - genome:queue — очередь на геномный взрыв
  TTL: 24h с продлением

L2: Edge LLM (1.5B GGUF) — 1.5GB, ~500ms
  Технология: GGUF quantized, feature-gate
  Только на нодах с достаточной RAM
  Активация: автоматическая при L2-L3 miss

L3: Центр Познаний (отдельный сервер 8-16GB)
  Технология: ASP-координатор + ChromaDB + LightRAG
  Содержит:
    - ASP-координатор (2GB, +10% hit rate)
    - ChromaDB (векторный архив ситуаций)
    - LightRAG (граф связей между ситуациями)
    - Redis master (все кэши 6 нод)
    - Nightly compilation engine
```

### 7.2 Redis DB Layout

| DB | Назначение | TTL |
|----|-----------|-----|
| 0  | llm:cache — кэш ответов LLM | 24h |
| 1  | asp:shared — общие ситуации | 7d |
| 2  | task:state — задачи | 48h |
| 3  | genome:queue — геномные очереди | до burst |
| 4  | confidence:map — confidence по ситуациям | 7d |
| 5  | prion:log — лог прион-распространений | 24h |
| 6  | metrics:kpi — метрики ноды | 30d |

## 11. Вопросы и риски

### Риски

| Риск | Вероятность | Смягчение |
|------|------------|-----------|
| SDM (Kanerva) сложен в реализации | Средняя | Начать с HashMap + Situator, SDM — v2 |
| Edge LLM 1.5B > 500ms на 2 ядрах | Низкая | TinyLlama 1.1B или fallback на ASP |
| 1GB на 167 не хватит для ASP | Средняя | ASP Mini: 200MB, без Edge LLM |
| Прион-шум (ложные ситуации) | Средняя | Bongard фильтр + Federator weight |
| Геномный взрыв не влезет в 15 мин | Низкая | Увеличить окно или дедуплицировать сильнее |

### Открытые вопросы

1. **SDM** — реализовывать полную разреженную память Kanerva или начать с упрощённого HashMap + ситуационный индекс? Рекомендация: HashMap v1, SDM v2.
2. **Bongard** — как именно реализовать верификацию? Через компиляцию кода (rustc --check) + тесты? Или через вторую LLM-проверку?
3. **Центр Познаний** — 8GB хватит или брать 16GB сразу? 8GB для старта, 16GB если ChromaDB + LightRAG + Redis + ASP живут вместе.
