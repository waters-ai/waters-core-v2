# Виртуальные режимы жизни агентов

> Как агенты живут 24/7: смена, питание, сон, связь, работа.
> waters-node + Kafka = кровеносная система роя.

---

## 1. Онтология жизни агента

```
ЖИЗНЬ АГЕНТА
├── Пробуждение (agent_open)
│   ├── Получение личности (тамагоча-память)
│   ├── Загрузка навыка (skill.json)
│   └── Подключение мостов (bridges.json)
│
├── Бодрствование (L0-L2)
│   ├── Получение задачи (Kafka: orders.v1)
│   ├── Работа: сбор/анализ/синтез
│   ├── Коммуникация (чат, голос, мнения)
│   └── Публикация находок (Kafka: findings.v1)
│
├── Сон (L3-L4)
│   ├── Сохранение состояния (Redis)
│   ├── Офлайн-очередь (невыполненные задачи)
│   └── Heartbeat (Kafka: heartbeat.v1)
│
└── Завершение (agent_close)
    ├── Сохранение журнала
    ├── Оценка (рейтинг)
    └── Слияние или архивация
```

## 2. Четыре времени жизни

### 2.1 Утро — Пробуждение и зарядка

```
[06:00-08:00]
Режим: L2 → L1 → L0
Питание: заряд батарей (солнце/сеть)
Kafka: DTN-буфер → flush → online

Что делают агенты:
  - power-manager: оценка энергии на день
  - coordinator: план задач на смену
  - все агенты: heartbeat → "готов к работе"
```

### 2.2 День — Полная активность

```
[08:00-18:00]
Режим: L0
Питание: пик (солнце/сеть)
Kafka: online, полный дуплекс

Что делают агенты:
  - field-агенты: поиск, сбор, первичный анализ
  - lab-агенты: анализ образцов,спектры
  - analytical-агенты: синтез, отчёты
  - coordinator: распределение задач
```

### 2.3 Вечер — Обработка и синхронизация

```
[18:00-22:00]
Режим: L1 → L2
Питание: снижение (закат/вечерний тариф)
Kafka: DTN-сжатие, bulk send

Что делают агенты:
  - synthesizer: отчёт дня
  - archivist: LightRAG ← ChromaDB
  - reporter: дашборды, презентации
  - coordinator: план на завтра
```

### 2.4 Ночь — Минимальное потребление

```
[22:00-06:00]
Режим: L3 → L4
Питание: батареи/резерв
Kafka: только heartbeat.v1 (каждые 5 мин)

Что делают агенты:
  - power-manager: мониторинг заряда
  - logger: запись метрик в offline queue
  - alert-агенты: только критическое
```

## 4. Kafka как сердечный ритм

Каждый агент имеет пульс — heartbeat в Kafka:

```json
{
  "agent_id": "navigator.1",
  "node_id": "node-1-atacama",
  "timestamp": "2026-05-16T14:00:00Z",
  "status": "working",
  "mode": "L0",
  "battery_pct": 85,
  "task_id": "ord_017",
  "findings_today": 7,
  "uptime_seconds": 21600
}
```

### Ритмы сердца агента

| Ритм | Частота | Топик | Назначение |
|------|---------|-------|-----------|
| 🟢 Спокойный | Каждые 60 с | heartbeat.v1 | "Я жив, всё нормально" |
| 🟡 Рабочий | Каждые 30 с | heartbeat.v1 | "Активно работаю" |
| 🟠 Нагрузка | Каждые 10 с | heartbeat.v1 | "Высокая нагрузка, нужны ресурсы" |
| 🔴 Авария | Каждые 5 с | alerts.v1 | "Критическая ситуация" |
| ⚫ Тишина | 3+ мин нет heartbeat | alerts.v1 | "Агент потерян" |

### Диагностика по пульсу

```python
def diagnose_agent(heartbeats: list[dict]) -> str:
    """По последовательности heartbeat определяет состояние"""
    if len(heartbeats) < 3:
        return "just_spawned"

    statuses = [h["status"] for h in heartbeats[-10:]]
    modes = [h["mode"] for h in heartbeats[-10:]]

    if "error" in statuses:
        return "degraded"
    if all(m == "L4" for m in modes):
        return "sleeping"
    if all(m == "L0" for m in modes):
        return "fully_active"
    if all(m in ["L2", "L3"] for m in modes):
        return "offline_mode"
    if statuses.count("working") > 7:
        return "productive"

    return "steady"
```

### Формула расхода энергии агента

```
E_agent = E_base + ∑(E_bridge × active) + ∑(E_llm × n_queries) + E_heartbeat

Где:
  E_base     = 5 Вт (Rust runtime, idle)
  E_bridge   = 2-10 Вт (MCP-клиент, NDI, Redis)
  E_llm      = 1-50 Вт (LLM-запрос, зависит от модели)
  E_heartbeat= 0.01 Вт (Kafka publish, раз в 60 с)
```

## 6. DTN-коммуникация между агентами

### 6.1 Восемь режимов общения через DTN

Каждый агент на каждой площадке живёт в одном из DTN-режимов. От этого зависит, как он общается с другими агентами.

```
Режим агента    Как общается
─────────────────────────────────────────────────
DTN-OFF     ↔ Полный дуплекс, реальное время
DTN-EVENT   → Дублирует важное в локальную очередь
DTN-STORE   → Копит всё, ничего не отправляет
DTN-CHUNK   → Режет большие данные на куски
DTN-PRIORITY→ Отправляет только важное (findings)
DTN-SCHEDULE→ Отправляет по расписанию
DTN-COMPRESS→ Сжимает перед отправкой
DTN-BEACON  → Только пульс, ничего не принимает
```

### 6.2 Матрица: агент A ↔ агент B через DTN

```
Как общаются два агента в разных DTN-режимах:

         B:OFF  B:EVENT B:STORE B:CHUNK B:PRIOR B:SCHED B:COMPR B:BEACON
A:OFF    ◇real   ◇real   ▷queue  ▷queue  ▷queue  ▷queue  ▷queue  ▷dead
A:EVENT  ◇real   ◇real   ▷queue  ▷queue  ▷queue  ▷queue  ▷queue  ▷dead  
A:STORE  ◇queue  ◇queue  ⬜none  ⬜none  ⬜none  ⬜none  ⬜none  ⬜dead
A:CHUNK  ◇chunk  ◇chunk  ⬜none  ⬜none  ⬜none  ⬜none  ⬜none  ⬜dead
A:PRIOR  ◇prior  ◇prior  ⬜none  ⬜none  ⬜none  ⬜none  ⬜none  ⬜dead
A:SCHED  ◇sched  ◇sched  ⬜none  ⬜none  ⬜none  ⬜none  ⬜none  ⬜dead
A:COMPR  ◇compr  ◇compr  ⬜none  ⬜none  ⬜none  ⬜none  ⬜none  ⬜dead
A:BEACON ▷dead   ▷dead   ⬜dead  ⬜dead  ⬜dead  ⬜dead  ⬜dead  ●alive

◇ = двусторонняя связь
▷ = односторонняя (A → B)
⬜ = нет связи
● = только heartbeat (оба знают, что живы)
```

### 6.3 Сквозные DTN-сценарии

#### Сценарий 1: Дрон нашёл метеорит, нет связи с MCC

```
Поле (Атакама, L2 → DTN-STORE)
  drone-operator: "confidence 0.95, mass 2.5 kg"
  → finding_id: f_023
  → offline_queue.push(f_023)

Лаборатория (Сантьяго, L0 → DTN-OFF)
  analyst: "жду новые образцы"
  → heartbeat: "жду"
  → НЕТ СВЯЗИ С ПОЛЕМ

Штаб (Москва, L0 → DTN-OFF)
  coordinator: "как прогресс?"
  → no finding from field 1h
  → alert: "node-1-atacama offline"

MCC (238, L0 → DTN-OFF)
  Архитектор: "ждём findings"
  → heartbeat поля: L2, battery 65%
  → решение: ждать рассвета

-- 4 часа спустя, восход --

Поле (Атакама, L2→L0, DTN-STORE→DTN-OFF)
  → offline_queue.flush()
  → findings: f_023 (meteorite, 2.5 kg, conf 0.95)
  → telemetry: 4h траекторий (сжато gzip)  
  → Kafka: flush завершён

MCC (238)
  → findings.f_023 получен
  → Архитектор: "подтверждаю метеорит"
  → orders: "ord_019 → lab: полный анализ"

Лаборатория (Сантьяго)
  → orders.ord_019 получен
  → analyst: начинает спектральный анализ
```

#### Сценарий 2: Лаборатория и поле обмениваются данными через DTN-PRIORITY

У полевой ноды мало энергии, но связь есть. Она отправляет только самое важное.

```
Поле (battery 25%, L2 → DTN-PRIORITY)
  Очередь:
    [1] findings: f_024 (meteorite, conf 0.92)
    [1] findings: f_025 (meteorite, conf 0.87)
    [2] heartbeat: "battery 25%, solar 10W"
    [3] telemetry: 200 KB
    [4] raw: спектр 50 MB
    [4] raw: LiDAR 120 MB

  Отправлено (за 30 сек окна связи):
    ✓ f_024
    ✓ f_025
    ✓ heartbeat
    ✗ telemetry (не хватило времени)
    ✗ raw (остались в очереди)

Лаборатория (L0 → DTN-OFF)
  Получено:
    ✓ findings → analyst → анализ
    ✓ heartbeat → "поле живёт, но устало"
  
MCC (238)
  → findings получены → Архитектор
  → решение: завтра с утра запросить telemetry
```

#### Сценарий 3: Туристическая группа в горах (все в разных DTN)

```
Турист 1 (вершина, L4 → DTN-BEACON)
  → heartbeat: "жив, -5°C, battery 12%"
  → SOS trigger: нет (пока)

Турист 2 (седловина, L3 → DTN-SCHEDULE)  
  → расписание: каждый час проверять связь
  → heartbeat туриста 1 получен: "жив"
  → store: свои треки, фото

Турист 3 (лагерь, L2 → DTN-STORE)
  → копит: треки группы, погоду, фото
  → ждёт расписания для flush

Турист 4 (базовый лагерь, L0 → DTN-OFF)
  → Starlink → Kafka
  → координатор группы

Когда всё синхронизируется:
  Турист 3 (flush) → Турист 4 (Kafka) → MCC
  → "группа жива, все heartbeats зелёные"
  → "найдено 2 потенциальных метеорита"
  → "Турист 1 нуждается в батареях завтра"
```

### 6.4 Агенты, живущие через DTN

У каждого агента есть навык работы в DTN — dtn_skill:

```rust
pub struct AgentDtnSkill {
    pub mode: DtnMode,           // текущий DTN-режим
    pub offline_queue: VecDeque<QueuedEvent>,
    pub max_queue_size: usize,   // лимит очереди
    pub priority_map: HashMap<String, u8>, // приоритеты топиков
    pub schedule: Vec<DtnSlot>,  // расписание (для DTN-SCHEDULE)
    pub last_flush: DateTime<Utc>,
}
```

Пример работы DTN-навыка у агента `drone-operator`:

```rust
fn on_finding(f: &Finding) {
    // 1. Сохранить в Redis (быстрый доступ)
    redis_hset(format!("finding:{}", f.id), serde_json::to_string(&f));
    
    // 2. Если есть связь — отправить в Kafka
    if self.dtn.mode == DtnMode::Off {
        kafka_send("findings", f);
    }
    
    // 3. Всегда — сохранить в offline queue (на случай обрыва)
    self.dtn.offline_queue.push_back(QueuedEvent {
        topic: "findings".into(),
        key: f.id.clone(),
        value: serde_json::to_vec(&f),
        priority: match f.confidence {
            c if c > 0.9 => 1,
            c if c > 0.7 => 2,
            _ => 3,
        },
        timestamp: Utc::now(),
    });
}
```

### 6.5 DTN-мосты между секторами жизни

```
Сектор A (DTN-STORE) ──► DTN-мост ◄── Сектор B (DTN-OFF)

DTN-мост — агент, который работает на стыке двух сред:
  - Понимает DTN-протокол
  - Буферизует данные с обеих сторон
  - Синхронизирует по расписанию
```

| Мост | Между | Что передаёт |
|------|-------|-------------|
| `field-lab-bridge` | Полигон (L2) ↔ Лаборатория (L0) | Образцы, спектры, координаты |
| `lab-mcc-bridge` | Лаборатория (L0) ↔ MCC (L0) | Отчёты, findings, запросы |
| `field-tourist-bridge` | Полигон (L2) ↔ Туризм (L3) | Карты, погода, опасности |
| `family-office-bridge` | Дом (L4 ночью) ↔ Офис (L0) | Планы, закупки, бюджет |

## 7. Коммуникация между агентами в разных времени жизни (Online)

### Оба в L0
```
Агент A ──Kafka findings──► Агент B
Агент B ──Kafka orders────► Агент A
Задержка: < 100 мс
Режим: полный диалог
```

### A в L2, B в L0
```
Агент B ──orders──► Kafka ──stores──► Агент A (offline queue)
Агент A ──findings──► local DTN буфер ──(flush при L0)──► Kafka
Задержка: минуты-часы
Режим: асинхронный
```

### Оба в L4
```
Агент A ──heartbeat──────► Kafka
Агент B ──heartbeat──────► Kafka
Никакой другой связи.
Задержка: ∞ (до пробуждения)
Режим: спячка
```
